Your tracing checklist (use this every time)
  1. Find the URL route in urls.py (or router registration).
  2. Find the view that handles it (CBV or ViewSet).
  3. List authentication options and the required permissions.
  4. Write down the queryset scoping rules.
  5. Check if it calls a service layer function.
  6. Predict the response shape and the error shape.
Mindset

How to reason without running code

  • Trust the code, not your guesses.
  • Security rules usually live in: permissions, queryset scoping, and services.
  • If a queryset is scoped, cross-tenant access often becomes a 404.
  • If a permission fails, you usually get a 401 or 403.
  • State changes usually go through a state machine.
This repo is security-heavy on purpose. That is good. It also means there are many layers. You will learn by naming each layer.

Tenant

Tenant isolation in web (URL company_pk)

A tenant is one company in a multi-company system.

For web views, tenant comes from the URL. The key is company_pk.

Codeapps/month_end/views.py
class MonthEndCompanyMixin:
    """Resolve company from URL and scope querysets to tenant."""

    def get_company(self):
        if not hasattr(self, "_company"):
            self._company = get_object_or_404(
                Company, pk=self.kwargs["company_pk"], is_active=True
            )
        return self._company

    def get_queryset(self):
        return MonthEndRun.objects.for_tenant(self.get_company())
  • If you change company_pk, you point at a different company.
  • Because the queryset is already scoped, cross-company objects often look like a 404.
Tenant

Tenant isolation in API (request.user.company and permissions)

For the API, tenant is tied to the logged-in user.

Scoping happens in a mixin and also in permissions.

Codeapps/api/views.py
class TenantScopedViewSetMixin:
    """Mixin to scope querysets to the authenticated user's tenant."""
    
    def get_queryset(self):
        """Filter queryset by user's company."""
        qs = super().get_queryset()
        user = self.request.user
        
        if not user or not user.is_authenticated:
            return qs.none()
        
        # Scheme admins with a company see only that company's data
        # Scheme admins without a company see all data (superusers)
        if user.company:
            return qs.filter(company=user.company)
        
        return qs
Notice the last line. If user.company is missing, the queryset is not filtered. This shows up again in the risk exercises.

class MonthEndRunDetailView(
    SchemeAdminRequiredMixin, MonthEndCompanyMixin, DetailView
):
    """View a single month-end run with its vesting events."""

    model = MonthEndRun
    template_name = "month_end/detail.html"
    context_object_name = "run"
    pk_url_kwarg = "pk"

    def get_context_data(self, **kwargs):
        ctx = super().get_context_data(**kwargs)
        ctx["vesting_events"] = self.object.vesting_events.select_related(
            "beneficiary"
        ).order_by("beneficiary__last_name", "beneficiary__first_name")
        ctx["tax_directives"] = self.object.tax_directives.select_related(
            "beneficiary"
        ).order_by("beneficiary__last_name")
        ctx["process_form"] = ProcessRunForm()
        return ctx

Codeapps/month_end/views.py
class MonthEndRunProcessView(_StateChangeView):
    """Process an APPROVED run — creates vesting events."""

    http_method_names = ["post"]

    def post(self, request, *args, **kwargs):
        run = self._get_run()
        form = ProcessRunForm(request.POST)

        if not form.is_valid():
            messages.error(request, "Invalid share price or tax rate.")
            return redirect(self._detail_url(run))

        try:
            process_run(
                run,
                user=request.user,
                share_price=form.cleaned_data["share_price"],
                tax_rate=form.cleaned_data["tax_rate"],
                ip_address=get_client_ip(request),
            )
            messages.success(
                request,
                f"Month-end run '{run.title}' processed — "
                f"{run.vesting_event_count} events created.",
            )
        except InvalidStateTransition as exc:
            messages.error(request, str(exc))
        except Exception:
            messages.error(
                request,
                f"Month-end run '{run.title}' failed. See details for more info.",
            )
        return redirect(self._detail_url(run))
  • Input validation is done by ProcessRunForm. (Form code is not shown.)
  • It calls process_run in the service layer.
  • It does not return JSON. It redirects and uses Django messages.
The broad except Exception: shows a message but does not log the exception here. This is one of the risks you will spot later.

Why

Why the service layer exists (and what it protects)

The service layer is where business rules live.

  • It enforces a state machine.
  • It writes audit logs.
  • It makes processing idempotent.
  • It uses atomic transactions.
See month-end service rules (state change and failure handling)
Codeapps/month_end/services.py
def _validate_transition(run: MonthEndRun, new_status: str) -> None:
    """Check that the transition is allowed by the state machine."""
    allowed = VALID_TRANSITIONS.get(run.status, [])
    if new_status not in allowed:
        raise InvalidStateTransition(
            f"Cannot transition from '{run.get_status_display()}' "
            f"to '{MonthEndRunStatus(new_status).label}'. "
            f"Allowed: {[MonthEndRunStatus(s).label for s in allowed]}"
        )

@transaction.atomic
def process_run(...):
    ...
    try:
        _create_vesting_events(...)
        ...
        run = _change_status(... MonthEndRunStatus.COMPLETED, ...)
    except Exception as exc:
        logger.exception("Month-end run %s failed: %s", run.pk, exc)
        run.failure_reason = str(exc)
        run.status = MonthEndRunStatus.FAILED
        run.save(update_fields=["status", "failure_reason", "updated_at"])
        raise
  • VALID_TRANSITIONS is the rule table. It lives in models and is not shown.
  • Errors are logged in the service. The run is marked FAILED.

Trace

Step-by-step API trace (from URL to JSON)

  1. Client hits /api/v1/....
  2. Router picks a ViewSet and an action (list, retrieve, create, or a custom action).
  3. DRF runs authentication (Token then Session).
  4. DRF runs permission checks.
  5. The ViewSet builds a scoped queryset (tenant logic).
  6. The serializer validates or formats data. (Serializer code is not shown.)
  7. The view returns a JSON response or raises an exception.
  8. Exceptions are wrapped by the custom exception handler into a standard envelope.

Codeapps/api/views.py
@action(detail=True, methods=["post"])
def process(self, request, pk=None):
    """Process an approved month-end run."""
    run = self.get_object()
    share_price = request.data.get("share_price")
    tax_rate = request.data.get("tax_rate")

    if not share_price or not tax_rate:
        return Response(
            {"error": {"code": "missing_parameters", "message": "share_price and tax_rate are required"}},
            status=status.HTTP_400_BAD_REQUEST,
        )

    try:
        with transaction.atomic():
            month_end_services.process_run(
                run=run,
                user=request.user,
                share_price=share_price,
                tax_rate=tax_rate,
            )
        return Response({"status": "processing"})
    except (ValueError, MonthEndInvalidStateTransition) as e:
        raise InvalidStateTransitionError(str(e))
  • This returns a JSON error envelope directly for missing parameters.
  • Invalid state transitions become a custom exception with a standard error envelope.
Why transaction.atomic() matters here

There are three layers that push toward atomic behavior:

  • Database setting: ATOMIC_REQUESTS wraps each request in a transaction.
  • API actions use with transaction.atomic().
  • Service functions are decorated with @transaction.atomic.
Codeconfig/settings/base.py
DATABASES["default"]["ATOMIC_REQUESTS"] = True

This reduces partial updates. If processing fails, you want a clean rollback.

Part 4

Permissions and tenant access (concrete)

DRF

Two levels: view-level vs object-level

View-level checks run before the view does real work.

Object-level checks run when DRF has an object.

Simple mental model (no magic words)
  1. DRF runs has_permission() for each permission class.
  2. If it is a detail action, DRF gets the object and runs has_object_permission() for each class.
  3. If any check fails, access is denied.

List endpoints often do not run object-level checks for each row. So queryset filtering becomes very important.

Key idea

Default-deny posture

  • If you are not authenticated, you should not see anything.
  • If you are authenticated but wrong role, you should be blocked.
  • If you are correct role but wrong tenant, you should not see cross-tenant data.

In this codebase, tenant isolation is enforced by both:

  • Queryset scoping in TenantScopedViewSetMixin.get_queryset()
  • Tenant access checks in permission classes
Permission code (read this like rules)
Codeapps/api/permissions.py
from rest_framework import permissions

from apps.accounts.models import UserRole


class IsTenantMember(permissions.BasePermission):
    ...
    def has_object_permission(self, request, view, obj):
        user = request.user
        
        # Must be authenticated
        if not user or not user.is_authenticated:
            return False
        
        # Get the resource's company
        resource_company = getattr(obj, "company", None)
        if resource_company is None:
            # If object has no company, allow (shouldn't happen for tenant-scoped models)
            return True
        
        return user.can_access_company(resource_company)


class IsSchemeAdmin(permissions.BasePermission):
    ...
    def has_permission(self, request, view):
        user = request.user
        return user and user.is_authenticated and user.role == UserRole.SCHEME_ADMIN


class IsBeneficiary(permissions.BasePermission):
    ...
    def has_permission(self, request, view):
        user = request.user
        return user and user.is_authenticated and user.role == UserRole.BENEFICIARY


class IsSchemeAdminOrReadOnly(permissions.BasePermission):
    def has_permission(self, request, view):
        user = request.user
        if not user or not user.is_authenticated:
            return False
        
        # Read permissions for any authenticated user
        if request.method in permissions.SAFE_METHODS:
            return True
        
        # Write permissions only for scheme admins
        return user.role == UserRole.SCHEME_ADMIN


class IsOwnerOrSchemeAdmin(permissions.BasePermission):
    ...
    def has_object_permission(self, request, view, obj):
        user = request.user
        
        if not user or not user.is_authenticated:
            return False
        
        # Scheme admins can access all (within tenant)
        if user.role == UserRole.SCHEME_ADMIN:
            # Still check tenant access
            resource_company = getattr(obj, "company", None)
            if resource_company:
                return user.can_access_company(resource_company)
            return True
        
        # Beneficiaries can only access their own records
        if user.role == UserRole.BENEFICIARY:
            # Check if this is a Beneficiary object
            if hasattr(obj, "user"):
                return obj.user == user
            # Check if this is an allocation/event linked to a beneficiary
            if hasattr(obj, "beneficiary"):
                beneficiary = obj.beneficiary
                return beneficiary.user == user if beneficiary else False
        
        return False


class TenantScopedPermission(permissions.BasePermission):
    ...
    def has_permission(self, request, view):
        user = request.user
        return user and user.is_authenticated
    
    def has_object_permission(self, request, view, obj):
        user = request.user
        
        if not user or not user.is_authenticated:
            return False
        
        # Get the resource's company
        resource_company = getattr(obj, "company", None)
        if resource_company is None:
            return True
        
        return user.can_access_company(resource_company)

State machines and idempotency

State machine

VALID_TRANSITIONS and _validate_transition

The service layer uses a state machine.

The rule table is VALID_TRANSITIONS. It is imported from models and is not shown.

Codeapps/month_end/services.py
def _validate_transition(run: MonthEndRun, new_status: str) -> None:
    """Check that the transition is allowed by the state machine."""
    allowed = VALID_TRANSITIONS.get(run.status, [])
    if new_status not in allowed:
        raise InvalidStateTransition(
            f"Cannot transition from '{run.get_status_display()}' "
            f"to '{MonthEndRunStatus(new_status).label}'. "
            f"Allowed: {[MonthEndRunStatus(s).label for s in allowed]}"
        )

    allowed = VALID_TRANSITIONS.get(run.status, [])
    if new_status not in allowed:
        raise InvalidStateTransition(...)
Four-eyes

Approver must not be the creator

This is a security control.

It blocks one person from creating and approving the same run.

Codeapps/dividends/services.py

        "The approver must be different from the creator (four-eyes principle)."
    )

Risk

12 spot-the-risk exercises (answers hidden)

Each risk is based only on the snippets.

Risk 1: API mixin returns unfiltered queryset if user.company is missing

What is the risk? What is a mitigation?

Answer

Risk: data leak across tenants. Mitigation: enforce company on all users or filter to none when company is missing, unless you have a clearly defined superuser rule.

Risk 2: IsTenantMember allows objects with no company

Why is return True on missing company dangerous?

Answer

If a tenant-owned model accidentally has company=None, the permission check would allow access.

Risk 3: TenantScopedPermission also allows objects with no company

What kind of bug could turn into a data leak?

Answer

A bug that creates rows without a company could bypass tenant checks. The code returns True when resource_company is None.

Risk 4: JSECompanySearchViewSet is not tenant-scoped

Why can this be okay, and when is it not okay?

Answer

It can be okay if it is truly reference data shared by all tenants. It is not okay if it contains tenant-sensitive data.

Risk 5: Broad except in web process view without logger.exception

What is the debugging risk?

Answer

You lose stack traces at the view layer. The service logs exceptions, but view-level context could be lost. It can also hide repeated failures.

Risk 6: API perform_create audit_action fallback

Spot what is odd about this default:

Codeapps/api/views.py
audit_action = action_map.get(model.__name__, AuditAction.BENEFICIARY_CREATE)
Answer

Unknown models would be logged as BENEFICIARY_CREATE. That can make the audit trail misleading.

Risk 7: API MonthEndRunViewSet.process passes share_price and tax_rate without type conversion

Service expects Decimal. API reads from request.data.

Answer

Risk: type errors or wrong math. Mitigation: validate and convert input in serializer or explicitly cast to Decimal with clear validation errors.

Risk 8: TenantScopedViewSetMixin comment says company-less scheme admins see all data

What must be true for this to be safe?

Answer

Only trusted superusers should be allowed to have no company. Otherwise it becomes a privilege escalation path.

Risk 9: failure_reason stores str(exc)

What is the info leak risk?

Answer

Error strings can include sensitive details. If shown to users, it can leak internals. Consider sanitizing or storing a safe message.

Risk 10: TenantScopedPermission.has_object_permission allows company=None

How would you detect this kind of issue early?

Answer

Add database constraints and model validation to prevent tenant-owned models from having company=None. Add tests that create and check access.

Risk 11: JSE enrich action queues a Celery task

What resource risk exists?

Answer

Task queue spam. Mitigation: add throttling, permissions, and audit logging around task triggers.

Risk 12: For list endpoints, object-level permissions may not run per item

Why does that matter?

Answer

If you rely on has_object_permission but do not filter the queryset, you may leak data in list responses. This repo adds queryset filters for beneficiaries in list endpoints.
