Part 1: Repo Map and Mental Model

Config vs Apps vs Templates

Django projects are organized into folders with specific purposes.

  • config/ - holds settings and top-level URL routing. The file config/settings/base.py defines security rules and middleware.
  • apps/ - contains features. Each folder inside (like apps/month_end/ or apps/api/) is called a Django app. Apps have models (database tables), views (logic), and other files.
  • templates/ - stores HTML files. Django renders these with data from views.
Key insight: Django separates configuration (config), business logic (apps), and presentation (templates).

Two Interfaces: Web and API

This codebase serves two types of users.

Web Interface (Class-Based Views)

Web users visit pages in a browser. They see HTML templates.

Django uses Class-Based Views (CBVs) to handle these requests.

Example: apps/month_end/views.py has views like MonthEndRunListView that render HTML.

API Interface (Django REST Framework)

API users send JSON and receive JSON. They might be mobile apps or scripts.

Django uses Django REST Framework (DRF) for this.

Example: apps/api/views.py has ViewSets like DividendRunViewSet.

Remember: Web uses CBVs and returns HTML. API uses ViewSets and returns JSON.

Tenant Isolation: Two Different Mechanisms

Tenant isolation means Company A cannot see Company B's data.

This codebase enforces isolation differently for web and API.

Web: URL-Based Scoping

Web URLs include company_pk (the company's ID).

Example URL pattern (not shown in snippets): /companies/<company_pk>/month-end-runs/

The view loads the company from company_pk and scopes all queries to that company.

# From apps/month_end/views.py
class MonthEndCompanyMixin:
    def dispatch(self, request, *args, **kwargs):
        self.company = get_object_or_404(Company, pk=kwargs["company_pk"])
        return super().dispatch(request, *args, **kwargs)

    def get_queryset(self):
        return MonthEndRun.objects.for_tenant(self.company)

This mixin loads the company from the URL and filters all queries by it.

API: User-Based Scoping

API requests do not have company_pk in the URL.

Instead, the API reads request.user.company.

Every user belongs to one company. Their API calls only see their company's data.

# From apps/api/views.py
class TenantScopedViewSetMixin:
    def get_queryset(self):
        qs = super().get_queryset()
        if hasattr(self.request.user, "company") and self.request.user.company:
            return qs.filter(company=self.request.user.company)
        # If user has no company, return unfiltered (admin users)
        return qs
Risk: If a user has no company, this returns the full queryset. We will explore this risk in exercises.

The API also uses object-level permissions to double-check access.

Part 2: Trace a Request End to End (Web)

Step-by-Step: GET Request (Viewing a List)

Let's trace what happens when a user visits a page showing month-end runs.

1. URL Matching - Django matches the URL to a route in urls.py (not shown in snippets). The route points to a view and extracts company_pk from the URL.
2. View Dispatch - Django calls MonthEndRunListView.dispatch(). The MonthEndCompanyMixin runs first and loads the company.
3. Queryset Scoping - The view calls get_queryset(), which uses MonthEndRun.objects.for_tenant(self.company) to filter runs by the company.
4. Template Context - The view passes the filtered queryset to the template as object_list or a named variable.
5. Template Render - Django renders the HTML template with the context data and returns it to the browser.

Step-by-Step: POST Request (State Change)

Now let's trace what happens when a user approves a month-end run.

1. URL Matching - The POST request matches a route like /companies/<company_pk>/month-end-runs/<pk>/approve/.
2. View Dispatch - Django calls MonthEndRunApproveView.post(). The mixin loads the company and the run.
3. Service Layer Call - The view does NOT change the database directly. Instead, it calls approve_run(run, user) from the service layer.
4. State Transition - The service layer checks if the transition is valid, enforces business rules (like four-eyes approval), updates the database, and logs an audit log.
5. Feedback - The view uses Django messages to show success ("Run approved") and redirects the user back to the detail page.
# From apps/month_end/views.py (conceptual, actual code varies)
class MonthEndRunApproveView(MonthEndCompanyMixin, View):
    def post(self, request, company_pk, pk):
        run = get_object_or_404(MonthEndRun, pk=pk, company=self.company)
        try:
            approve_run(run, request.user)
            messages.success(request, "Month-end run approved successfully.")
        except InvalidStateTransitionError as e:
            messages.error(request, str(e))
        return redirect("month-end-run-detail", company_pk=company_pk, pk=pk)

Why the Service Layer Exists

The service layer is code in files like apps/month_end/services.py.

It contains business logic that is too complex for views.

The service layer protects you from:

  • Invalid state transitions - You cannot go from COMPLETED to DRAFT directly.
  • Business rule violations - You cannot approve your own month-end run (four-eyes principle).
  • Data corruption - Service functions use transaction.atomic() so failures roll back all changes.
  • Missing audit logs - Every state change is logged automatically.
Rule: Views handle HTTP concerns (parsing requests, rendering responses). Services handle business logic (validation, state changes, auditing).

Part 3: Trace a Request End to End (API)

Step-by-Step: GET Request (List Endpoint)

Let's trace what happens when an API client calls GET /api/v1/dividend-allocations/?run=123.

1. URL Routing - Django matches /api/v1/dividend-allocations/ to the router in apps/api/urls.py. The router maps it to DividendAllocationViewSet.list().
2. Authentication - DRF checks for a token in the Authorization: Token <token> header or a session cookie. If neither is present, the request is rejected (401).
3. View-Level Permission - DRF calls has_permission() on the permission classes. For this endpoint, it checks if the user is authenticated.
4. Queryset Scoping - The viewset calls get_queryset(). The TenantScopedViewSetMixin filters by request.user.company. Then the viewset applies the ?run=123 filter.
5. Object-Level Permission - DRF calls has_object_permission() on each allocation. Beneficiaries can only see their own allocations.
6. Serialization - DRF passes the filtered queryset to a serializer, which converts database objects to JSON.
7. Response - DRF returns a JSON response with the list of allocations.
# From apps/api/views.py
class DividendAllocationViewSet(TenantScopedViewSetMixin, viewsets.ReadOnlyModelViewSet):
    queryset = DividendAllocation.objects.all()
    serializer_class = DividendAllocationSerializer
    permission_classes = [IsAuthenticated, IsOwnerOrSchemeAdmin]

    def get_queryset(self):
        qs = super().get_queryset()
        run_id = self.request.query_params.get("run")
        if run_id:
            qs = qs.filter(run_id=run_id)
        # Beneficiaries only see their own allocations
        if self.request.user.role == UserRole.BENEFICIARY:
            qs = qs.filter(beneficiary__user=self.request.user)
        return qs

Step-by-Step: POST Request (Custom Action)

Let's trace what happens when an API client calls POST /api/v1/dividend-runs/123/approve/.

1. URL Routing - The router matches this to DividendRunViewSet.approve() because of the @action decorator.
2. Authentication - Same as above. Token or session required.
3. View-Level Permission - The @action specifies permission_classes=[IsSchemeAdmin], so only scheme admins can call this.
4. Object Retrieval - DRF calls get_object(), which retrieves the run with pk=123 and applies tenant scoping.
5. Object-Level Permission - DRF checks has_object_permission(). The tenant permission verifies the run belongs to the user's company.
6. Transaction Start - The action is wrapped in transaction.atomic(), so if anything fails, all changes roll back.
7. Service Layer Call - The action calls approve_run(run, request.user) from the service layer.
8. Audit Logging - The service layer logs the approval with log_audit().
9. Response - The action returns a JSON response with status 200 and a message like {"message": "Run approved successfully"}.
# From apps/api/views.py
class DividendRunViewSet(TenantScopedViewSetMixin, viewsets.ModelViewSet):
    @action(detail=True, methods=["post"], permission_classes=[IsSchemeAdmin])
    def approve(self, request, pk=None):
        run = self.get_object()
        with transaction.atomic():
            try:
                approve_run(run, request.user)
                return Response({"message": "Run approved successfully"})
            except InvalidStateTransitionError as e:
                return Response({"error": str(e)}, status=status.HTTP_400_BAD_REQUEST)

Custom Actions vs CRUD

DRF provides default CRUD operations (Create, Read, Update, Delete).

For example, POST /api/v1/dividend-runs/ creates a new run.

Custom actions use the @action decorator to add extra endpoints like approve, process, and reset.

Pattern: Use custom actions for state transitions and business operations that do not map to simple CRUD.

When transaction.atomic() Matters

The transaction.atomic() decorator wraps code in a database transaction.

If any line inside raises an exception, Django rolls back all changes.

Use it when:

  • You update multiple database records in one operation
  • You want all changes to succeed or all to fail (no partial updates)
  • You are changing state that affects money or compliance

Example: processing a dividend run creates hundreds of allocations. If allocation 50 fails, you want to undo allocations 1 through 49.

# From apps/dividends/services.py
@transaction.atomic()
def process_run(run: DividendRun) -> DividendRun:
    _validate_transition(run, RunStatus.PROCESSING)
    _change_status(run, RunStatus.PROCESSING)
    try:
        _create_allocations(run)
        totals = _compute_totals(run)
        run.total_shares = totals["shares"]
        run.total_dividend = totals["dividend"]
        run.processed_at = timezone.now()
        _change_status(run, RunStatus.COMPLETED)
        return run
    except Exception as e:
        logger.exception("Processing failed for run %s", run.pk)
        run.failure_reason = str(e)
        _change_status(run, RunStatus.FAILED)
        run.save()
        raise

If _create_allocations() fails, Django rolls back the status change to PROCESSING and the run stays in APPROVED state.

Part 4: Permissions and Tenant Access

has_permission vs has_object_permission

DRF checks permissions at two levels.

View-Level: has_permission

This runs before Django retrieves any objects.

It decides if the user can access the endpoint at all.

Example: IsAuthenticated checks if the user is logged in.

# From apps/api/permissions.py
class IsSchemeAdmin(BasePermission):
    def has_permission(self, request, view):
        return request.user and request.user.role == UserRole.SCHEME_ADMIN

Object-Level: has_object_permission

This runs after Django retrieves the object.

It decides if the user can access this specific object.

Example: IsTenantMember checks if the object belongs to the user's company.

# From apps/api/permissions.py
class IsTenantMember(BasePermission):
    def has_object_permission(self, request, view, obj):
        if not hasattr(obj, "company"):
            return False
        return request.user.can_access_company(obj.company)
Risk: If obj has no company attribute, this permission denies access. But what if the object should be accessible to everyone (like reference data)? We handle that by not applying IsTenantMember to reference endpoints.

Permission Scenarios

Let's trace 8 scenarios to understand how permissions combine.

Scenario 1: Scheme Admin Reading Own Company's Dividend Run

  • Request: GET /api/v1/dividend-runs/123/
  • User: Scheme admin at Company A
  • Object: Dividend run belonging to Company A
  • View-Level Permission: IsAuthenticated passes (user is logged in).
  • Queryset Scoping: TenantScopedViewSetMixin filters by Company A.
  • Object-Level Permission: TenantScopedPermission.has_object_permission() checks obj.company == user.company. Passes.
  • Result: ALLOW

Scenario 2: Scheme Admin Trying to Read Another Company's Dividend Run

  • Request: GET /api/v1/dividend-runs/456/
  • User: Scheme admin at Company A
  • Object: Dividend run belonging to Company B
  • View-Level Permission: Passes.
  • Queryset Scoping: The queryset filters by Company A, so run 456 is not in the queryset.
  • Object Retrieval: Django raises Http404 because run 456 is not found in the filtered queryset.
  • Result: DENY (404)
Key insight: Tenant scoping happens at the queryset level. If the object is not in the queryset, the request fails before object permissions run.

Scenario 3: Beneficiary Trying to Access Another Beneficiary's Allocation

  • Request: GET /api/v1/dividend-allocations/789/
  • User: Beneficiary Alice at Company A
  • Object: Allocation for Beneficiary Bob at Company A
  • View-Level Permission: IsAuthenticated passes.
  • Queryset Scoping: The viewset filters by company and then by beneficiary__user=request.user. Bob's allocation is not in the queryset.
  • Object Retrieval: Django raises Http404.
  • Result: DENY (404)

Scenario 4: Beneficiary Accessing Own Vesting Event

  • Request: GET /api/v1/vesting-events/111/
  • User: Beneficiary Alice at Company A
  • Object: Vesting event for Alice
  • View-Level Permission: IsAuthenticated passes.
  • Queryset Scoping: Filtered by company and beneficiary__user=request.user. Event 111 is in the queryset.
  • Object-Level Permission: IsOwnerOrSchemeAdmin.has_object_permission() checks if obj.beneficiary.user == request.user. Passes.
  • Result: ALLOW

Scenario 5: Unauthenticated Access

  • Request: GET /api/v1/beneficiaries/
  • User: Anonymous (no token or session)
  • View-Level Permission: IsAuthenticated.has_permission() fails.
  • Result: DENY (401)

Scenario 6: Reference Data Endpoint (JSE Companies)

  • Request: GET /api/v1/jse-companies/?q=sasol
  • User: Any authenticated user
  • Object: Reference data (JSE company records)
  • View-Level Permission: IsAuthenticated passes.
  • Queryset Scoping: JSECompanySearchViewSet does NOT inherit TenantScopedViewSetMixin. The queryset is not scoped by company.
  • Object-Level Permission: Not applied (this is reference data).
  • Result: ALLOW
Important: Not all data is tenant-scoped. Reference data like JSE company names is shared across all tenants. The viewset intentionally skips tenant scoping.

Scenario 7: Beneficiary Trying to Create a Dividend Run

  • Request: POST /api/v1/dividend-runs/
  • User: Beneficiary at Company A
  • View-Level Permission: IsSchemeAdminOrReadOnly.has_permission() checks the request method. POST is not a safe method, so it requires IsSchemeAdmin. User is a beneficiary, so the check fails.
  • Result: DENY (403)
# From apps/api/permissions.py
class IsSchemeAdminOrReadOnly(BasePermission):
    def has_permission(self, request, view):
        if request.method in SAFE_METHODS:
            return request.user and request.user.is_authenticated
        return request.user and request.user.role == UserRole.SCHEME_ADMIN

Scenario 8: Scheme Admin Creating a Dividend Run (User Has No Company)

  • Request: POST /api/v1/dividend-runs/ with {"name": "Q1 2024", "company": 999}
  • User: Scheme admin with user.company = None
  • View-Level Permission: IsSchemeAdmin passes.
  • perform_create: The viewset calls TenantScopedViewSetMixin.perform_create(). This method checks if request.user.company exists. If not, it raises TenantMismatchError.
  • Result: DENY (403)
# From apps/api/views.py
def perform_create(self, serializer):
    if not hasattr(self.request.user, "company") or not self.request.user.company:
        raise TenantMismatchError("User must belong to a company to create resources.")
    serializer.save(company=self.request.user.company, created_by=self.request.user)
    log_audit(
        user=self.request.user,
        action=AuditAction.CREATE,
        content_object=serializer.instance,
        extra=_make_json_safe(serializer.validated_data),
    )
Security note: The API prevents users without a company from creating or modifying tenant-scoped resources. This is critical for tenant isolation.

Part 5: Error Responses and Debugging

The JSON Error Envelope

The API returns errors in a consistent format.

This format is defined in apps/api/exceptions.py.

# From apps/api/exceptions.py
def custom_exception_handler(exc, context):
    response = exception_handler(exc, context)
    if response is not None:
        error_code = _get_error_code(exc)
        error_payload = {
            "error": {
                "code": error_code,
                "message": str(exc),
            }
        }
        if hasattr(exc, "detail") and isinstance(exc.detail, dict):
            error_payload["error"]["details"] = exc.detail
        response.data = error_payload
    return response

Every API error response looks like this:

{
  "error": {
    "code": "tenant_mismatch",
    "message": "User must belong to a company to create resources.",
    "details": {...}  // optional
  }
}

Part 6: State Machines and Idempotency

VALID_TRANSITIONS Concept

Both apps/month_end/services.py and apps/dividends/services.py define a state machine.

A state machine is a set of allowed transitions between states.

Example from the services:

# From apps/month_end/services.py and apps/dividends/services.py
VALID_TRANSITIONS = {
    RunStatus.DRAFT: [RunStatus.PENDING_APPROVAL],
    RunStatus.PENDING_APPROVAL: [RunStatus.APPROVED, RunStatus.DRAFT],
    RunStatus.APPROVED: [RunStatus.PROCESSING, RunStatus.DRAFT],
    RunStatus.PROCESSING: [RunStatus.COMPLETED, RunStatus.FAILED],
    RunStatus.COMPLETED: [RunStatus.DRAFT],
    RunStatus.FAILED: [RunStatus.DRAFT],
}

This dictionary maps each state to the states it can transition to.

Example: From APPROVED, you can go to PROCESSING or DRAFT.

You cannot go from APPROVED to COMPLETED directly.

From COMPLETED or FAILED, you can reset to DRAFT.

Four-Eyes Principle

The four-eyes principle means one person cannot both create and approve a run.

This prevents fraud and ensures accountability.

# From apps/month_end/services.py and apps/dividends/services.py
def approve_run(run, user):
    if run.created_by == user:
        raise PermissionDenied("You cannot approve a run you created.")
    _validate_transition(run, RunStatus.APPROVED)
    run.approved_by = user
    run.approved_at = timezone.now()
    _change_status(run, RunStatus.APPROVED)
    return run

If the user who created the run tries to approve it, the function raises PermissionDenied.

Idempotency Guard Patterns

Idempotency means running an operation multiple times has the same effect as running it once.

Example: If you process a dividend run twice, you should not create duplicate allocations.

# From apps/dividends/services.py
def _create_allocations(run: DividendRun):
    # Idempotency guard: exit if allocations already exist
    if DividendAllocation.objects.filter(run=run).exists():
        logger.info("Allocations already exist for run %s, skipping creation.", run.pk)
        return

    beneficiaries = Beneficiary.objects.filter(company=run.company, status=BeneficiaryStatus.ACTIVE)
    allocations = []
    for b in beneficiaries:
        allocation = DividendAllocation(
            run=run,
            beneficiary=b,
            shares=b.vested_shares,
            dividend_amount=b.vested_shares * run.dividend_per_share,
        )
        allocations.append(allocation)
    DividendAllocation.objects.bulk_create(allocations)

The guard checks if allocations already exist. If they do, the function exits early without creating duplicates.

Why Decimal Is Used

The codebase uses Python's Decimal type for money and share amounts.

This is because floating-point numbers (like 0.1 + 0.2) can have rounding errors.

Decimal ensures exact precision for financial calculations.

# Example (not from snippets, but illustrates the concept)
from decimal import Decimal

# BAD (floating point)
total = 0.1 + 0.2  # 0.30000000000000004

# GOOD (Decimal)
total = Decimal("0.1") + Decimal("0.2")  # Decimal('0.3')
Rule: Always use Decimal for money, tax rates, and share amounts. Never use float.

Glossary

Click a term in the guide to jump to its definition here.

Tenant Isolation

A security pattern where each company's data is kept separate from other companies.

In this codebase, tenant isolation is enforced by filtering queries by company.

Django App

A Python package inside the apps/ folder that contains models, views, and other code for a specific feature.

Examples: apps/month_end/, apps/dividends/, apps/api/.

Class-Based View (CBV)

A Django view written as a class instead of a function.

CBVs provide reusable patterns like ListView, DetailView, and CreateView.

Django REST Framework (DRF)

A library for building REST APIs in Django.

It provides ViewSets, serializers, and permission classes.

ViewSet

A DRF class that combines logic for multiple related endpoints (list, create, retrieve, update, delete).

Example: BeneficiaryViewSet handles all CRUD operations for beneficiaries.

Object-Level Permission

A permission check that runs after an object is retrieved.

It decides if the user can access this specific object.

Example: checking if obj.company == user.company.

Service Layer

Code that contains business logic separate from views and models.

In this codebase, service functions handle state transitions, validation, and audit logging.

Files: apps/month_end/services.py, apps/dividends/services.py.

Four-Eyes Principle

A control where one person cannot both create and approve the same record.

This prevents fraud and ensures accountability.

Audit Log

A record of who did what and when.

Every state change, creation, update, and deletion is logged for compliance and debugging.

Serializer

A DRF class that converts database objects to JSON and validates input.

Example: BeneficiarySerializer defines which fields are included in the API response.

1. What is the primary difference between the web interface and the API interface in this codebase?

Web uses Class-Based Views (CBVs) and returns HTML templates. API uses Django REST Framework ViewSets and returns JSON.

4. What happens if a user with no company tries to create a dividend run via the API?

The API raises TenantMismatchError in perform_create() and returns a 403 response with error code tenant_mismatch.

The service layer protects from:

JSE companies are reference data (publicly traded companies). They are not tenant-specific, so the endpoint intentionally skips tenant scoping. All authenticated users can access them.

11. What are the valid transitions from the APPROVED state?

From APPROVED, you can transition to PROCESSING (by calling process_run()) or DRAFT (by calling reset_to_draft()).

13. Why does the process_run() function use transaction.atomic()?

It ensures that all database changes (status updates, allocation creation, totals) either all succeed or all roll back. This prevents partial updates and data corruption.

15. When is CSRF protection required?

CSRF protection is required for session-based authentication (cookie). It is NOT required for token-based authentication because tokens are not stored in cookies.

Alice sees only her own allocation. The viewset filters by beneficiary__user=request.user when the user role is BENEFICIARY.

17. What does reset_to_draft() do to allocations or vesting events?

It deletes all allocations or vesting events associated with the run, resets totals to 0, and changes the status to DRAFT.

18. Why should you use Decimal instead of float for money?

Floats have rounding errors that can corrupt financial calculations. Decimal provides exact precision for money and tax rates.

19. What HTTP status code is returned when a user is not authenticated?

401 (Unauthorized) with error code not_authenticated.

20. What HTTP status code is returned when a user is authenticated but lacks permission to access a resource?

403 (Forbidden) with error code permission_denied or tenant_mismatch.

21. Name two custom actions (beyond CRUD) available on DividendRunViewSet.

approve, process, reset

22. A scheme admin creates a beneficiary with POST /api/v1/beneficiaries/ and includes "company": 999 in the request body (a different company). What company is the beneficiary created with?

The beneficiary is created with company=request.user.company. The request body is ignored. perform_create() overrides the company field server-side to prevent bypassing tenant isolation.

23. What security setting ensures session cookies are only sent over HTTPS?

SESSION_COOKIE_SECURE = True in settings (from config/settings/base.py).

24. If process_run() fails halfway through, what state is the run left in?

The run is set to FAILED status. The failure reason is stored in run.failure_reason. If transaction.atomic() is used, partial allocations are rolled back.

25. What is the purpose of the @action decorator in DRF ViewSets?

The @action decorator adds custom endpoints to a ViewSet beyond the standard CRUD operations. Examples include approve, process, and reset.
