Part 1: Repo map and mental model

We build a simple map in your head so you stop feeling lost in a big codebase.

Foundations
Three big folders to understand
  • config/: Django project configuration. Settings, root URLs, WSGI.
  • apps/: your business domains. Each app owns models, services, views, templates.
  • templates/: shared templates. Apps can also have their own templates.
This repo has two interfaces
  • Web pages: Django views (class-based views). They render HTML and use Django messages.
  • API: Django REST Framework (DRF). ViewSets return JSON and use permission classes.
Tenant isolation in this repo (two different paths)

Tenant here means "a company". Your first question in most code is: "Which company is this request for?"

  • Web path (month-end views): company comes from the URL as company_pk. The mixin loads the Company and then filters querysets using for_tenant(company).
  • API path (DRF): company comes from request.user.company. The API mixin filters querysets by that company.

If you miss this, you will misunderstand almost everything else.

The security mindset this code is aiming for
  • Default deny: no access unless you can prove you should have it.
  • Tenant scoping everywhere: you never want data from another company to leak.
  • Service layer for state changes: it makes state moves safe and auditable.
  • Audit logging: every important action should leave a trail.

Part 2: Trace a request end to end (web)

We trace how a browser request becomes HTML, and how POST actions change state safely.

CBVs
Typical web GET flow in this repo
  1. URL includes company_pk (not shown in snippets, but used by the views).
  2. MonthEndCompanyMixin.get_company() loads the Company.
  3. get_queryset() scopes to tenant: MonthEndRun.objects.for_tenant(company).
  4. CBV uses that queryset to fetch objects.
  5. get_context_data adds company and other context.
  6. Django renders a template.
Key snippet: tenant scoping in the web views 
class MonthEndCompanyMixin:
    """Resolve company from URL and scope querysets to tenant."""

    def get_company(self):
        if not hasattr(self, "_company"):
            self._company = get_object_or_404(
                Company, pk=self.kwargs["company_pk"], is_active=True
            )
        return self._company

    def get_queryset(self):
        return MonthEndRun.objects.for_tenant(self.get_company())

    def get_context_data(self, **kwargs):
        ctx = super().get_context_data(**kwargs)
        ctx["company"] = self.get_company()
        return ctx

Notice the pattern: company comes from the URL, not from the user object.

Typical web POST state-change flow
  1. The view is POST-only (http_method_names = ["post"]).
  2. The view loads the run from the tenant-scoped queryset.
  3. It validates a form (for process).
  4. It calls a service function like approve_run or process_run.
  5. It shows a message and redirects back to the detail page.
Key snippet: process view validates form then calls service 
class MonthEndRunProcessView(_StateChangeView):
    """Process an APPROVED run - creates vesting events."""

    http_method_names = ["post"]

    def post(self, request, *args, **kwargs):
        run = self._get_run()
        form = ProcessRunForm(request.POST)

        if not form.is_valid():
            messages.error(request, "Invalid share price or tax rate.")
            return redirect(self._detail_url(run))

        try:
            process_run(
                run,
                user=request.user,
                share_price=form.cleaned_data["share_price"],
                tax_rate=form.cleaned_data["tax_rate"],
                ip_address=get_client_ip(request),
            )
            messages.success(
                request,
                f"Month-end run '{run.title}' processed - "
                f"{run.vesting_event_count} events created.",
            )
        except InvalidStateTransition as exc:
            messages.error(request, str(exc))
        except Exception:
            messages.error(
                request,
                f"Month-end run '{run.title}' failed. See details for more info.",
            )
        return redirect(self._detail_url(run))

Two common Django ideas here: forms do validation, and redirects stop double submissions on refresh.

Why the service layer exists
  • It centralises business rules. You do not want rules scattered across views and admin pages.
  • It makes state transitions safe. A run should not jump to any status at any time.
  • It adds audit logging in one place. That keeps your trail complete.
  • It wraps changes in transaction.atomic so partial updates are less likely.
Concrete example: four-eyes principle is enforced in services 
@transaction.atomic
def approve_run(
    run: MonthEndRun,
    *,
    user,
    ip_address: str | None = None,
) -> MonthEndRun:
    """
    Move a DRAFT run to APPROVED.
    Requires a different user than the creator (four-eyes principle).
    """
    if user == run.created_by:
        raise InvalidStateTransition(
            "The approver must be different from the creator (four-eyes principle)."
        )

    run = _change_status(
        run, MonthEndRunStatus.APPROVED, user=user, ip_address=ip_address,
    )
    run.approved_by = user
    run.approved_at = timezone.now()
    run.save(update_fields=["approved_by", "approved_at", "updated_at"])
    return run

This rule is not optional. It is part of the process design.

Tenant scoping in the API (critical)

This mixin filters querysets by request.user.company.

class TenantScopedViewSetMixin:
    """Mixin to scope querysets to the authenticated user's tenant."""

    def get_queryset(self):
        """Filter queryset by user's company."""
        qs = super().get_queryset()
        user = self.request.user

        if not user or not user.is_authenticated:
            return qs.none()

        # Scheme admins with a company see only that company's data
        # Scheme admins without a company see all data (superusers)
        if user.company:
            return qs.filter(company=user.company)

        return qs
Risk to notice: if user.company is missing, this returns the full queryset. That can be fine only if that state is tightly controlled (for example superusers only).
You must treat this as a hot spot.
Write operations: company and created_by are set server-side 
def perform_create(self, serializer):
    """Set company and created_by from authenticated user."""
    user = self.request.user
    extra_kwargs = {}

    # Set company if the model has it
    model = serializer.Meta.model
    if hasattr(model, "company"):
        if not user.company:
            raise TenantMismatchError("You must be associated with a company to create records.")
        extra_kwargs["company"] = user.company

    # Set created_by if the model has it
    if hasattr(model, "created_by"):
        extra_kwargs["created_by"] = user

    instance = serializer.save(**extra_kwargs)

    ... log_audit(...)

    return instance

This prevents the client from forging the company on create. The server decides.

Custom actions: approve/process/reset are not CRUD

These are extra endpoints on a single object. DRF makes them with @action(detail=True, methods=["post"]).

@action(detail=True, methods=["post"])
def approve(self, request, pk=None):
    """Approve a dividend run (four-eyes principle)."""
    run = self.get_object()
    try:
        with transaction.atomic():
            dividend_services.approve_run(run=run, user=request.user)
        return Response({"status": "approved"})
    except (ValueError, DividendInvalidStateTransition) as e:
        raise InvalidStateTransitionError(str(e))

Part 4: Permissions and tenant access

This is where most security bugs live. We make it concrete using scenarios.

Access control
Two levels of checks
  • has_permission: view-level check. Runs before fetching objects.
  • has_object_permission: object-level check. Runs when you have a specific object.
Tenant check shape

Most checks end up calling user.can_access_company(company). The method is not shown here, but its result is the key.

Permission snippets to anchor your thinking 
class IsSchemeAdmin(permissions.BasePermission):
    """Only allow scheme admin users."""

    message = "This action requires scheme administrator privileges."

    def has_permission(self, request, view):
        user = request.user
        return user and user.is_authenticated and user.role == UserRole.SCHEME_ADMIN
class IsOwnerOrSchemeAdmin(permissions.BasePermission):
    message = "You can only access your own records."

    def has_object_permission(self, request, view, obj):
        user = request.user

        if not user or not user.is_authenticated:
            return False

        if user.role == UserRole.SCHEME_ADMIN:
            resource_company = getattr(obj, "company", None)
            if resource_company:
                return user.can_access_company(resource_company)
            return True

        if user.role == UserRole.BENEFICIARY:
            if hasattr(obj, "user"):
                return obj.user == user
            if hasattr(obj, "beneficiary"):
                beneficiary = obj.beneficiary
                return beneficiary.user == user if beneficiary else False

        return False
8 concrete scenarios (expected allow or deny)
Scenario Likely result Why (based on snippets)
Scheme admin reads a beneficiary in their company ALLOW Authenticated, role is scheme admin, tenant access passes.
Scheme admin tries to read a beneficiary from another company DENY Object-level tenant access should fail (user.can_access_company false). Also queryset scoping may prevent lookup.
Beneficiary requests allocations but tries to view another person’s allocation DENY DividendAllocationViewSet.get_queryset filters to beneficiary_profile for beneficiary users.
Beneficiary requests their own vesting events for a run ALLOW VestingEventViewSet.get_queryset filters to their own beneficiary_profile.
Unauthenticated user calls any normal API endpoint DENY Default DRF permission is authenticated only.
Unauthenticated user hits the token endpoint with correct credentials ALLOW Token endpoint is meant to issue tokens. (Auth validation handled by DRF token view.)
Any scheme admin uses JSE company search endpoint ALLOW Permission is scheme admin only. Not tenant-scoped on purpose because it is reference data.
Scheme admin tries POST approve on a run they created DENY Service layer enforces four-eyes and raises InvalidStateTransition.
Important: In DRF, permissions can be stacked. For example, a ViewSet might have both TenantScopedPermission and IsSchemeAdmin. Both must pass.
What to watch for (risk list)
  • Objects without company: IsTenantMember says “allow” if obj.company is missing. That might be fine for truly global tables, but it can also hide a modelling mistake.
  • user.company missing in API: mixin returns full queryset. That is a sharp edge.
  • Reference data endpoints: not tenant-scoped. That is only safe if the data is truly non-sensitive.

Part 5: Error responses and debugging

Every API error becomes one JSON shape. That is great for clients, and it changes how you debug.

Errors
The error envelope (exact shape)

When DRF handles an exception, this handler wraps the output like this:

{
  "error": {
    "code": "error_code",
    "message": "Human-readable message",
    "details": {...}   // optional
  }
}

That means your frontend should not expect DRF’s default structures.

Part 6: State machines and idempotency

Month-end and dividends both use the same pattern: validate transitions, run atomic, log audit, and avoid duplicates.

Services
State machine basics
  • A run has a status like DRAFT, APPROVED, PROCESSING, COMPLETED, FAILED.
  • You cannot jump anywhere you like. Valid moves are defined by VALID_TRANSITIONS.
  • Services enforce this with _validate_transition.
Idempotency in plain terms

If the same request runs twice, the second run should not create duplicates. This code checks for existing events or allocations and skips creation.

Visual state diagrams (simplified)
Month-end run
DRAFT → APPROVED → PROCESSING → COMPLETED
DRAFT → APPROVED → PROCESSING → FAILED
FAILED → DRAFT (reset)
APPROVED → DRAFT (reset)
Exact allowed moves live in VALID_TRANSITIONS (not shown here).
Dividend run
DRAFT → APPROVED → PROCESSING → COMPLETED
DRAFT → APPROVED → PROCESSING → FAILED
FAILED → DRAFT (reset)
APPROVED → DRAFT (reset)
Same idea, different domain.
Idempotency guard example (month-end) 
if run.vesting_events.exists():
    logger.info("Vesting events already exist for run %s - skipping.", run.pk)
    return 0

This makes repeated processing much safer.

Why Decimal is used

Money maths using floats can drift. Decimal keeps exact rounding rules.

gross = (Decimal(shares) * share_price).quantize(Decimal("0.01"))
tax = (gross * tax_rate).quantize(Decimal("0.01"))
net = gross - tax
What happens on failure (month-end) 
except Exception as exc:
    logger.exception("Month-end run %s failed: %s", run.pk, exc)
    run.failure_reason = str(exc)
    run.status = MonthEndRunStatus.FAILED
    run.save(update_fields=["status", "failure_reason", "updated_at"])
    raise

Important detail: it re-raises. That means the caller will also see an error unless it catches it.

Glossary

Click a term to jump to the definition. Terms are based on the guide content.

Reference
tenant router ViewSet CBV service layer transaction.atomic CSRF idempotency audit log permissions
tenant

A company boundary. Data from one tenant should not leak to another tenant.

router

A DRF helper that builds URL patterns from ViewSets. You register resources and it generates list and detail endpoints.

ViewSet

A DRF class that groups actions for a resource. It can support CRUD plus extra actions like approve and process.

CBV (class-based view)

A Django view built as a class. You inherit from ListView, DetailView, CreateView, UpdateView, or View and override methods.

service layer

Plain Python functions that contain business logic. Views call services, services validate and mutate, then log audit events.

transaction.atomic

A transaction wrapper. If an error happens, the DB changes inside the block roll back together.

CSRF

A browser protection for cookie-based sessions. It mainly matters for POST/PUT/PATCH/DELETE when using SessionAuthentication.

idempotency

If you run the same operation twice, you do not want duplicates. The code checks if events or allocations already exist and skips creation.

audit log

A record of who did what, when, and to which object. In finance workflows this is often required.

permissions

Checks that decide if a user can access a view and a specific object. In DRF you often have view-level and object-level checks.

"""
API Permission classes for tenant-scoped access control.

Security notes:
- All permissions are default-deny.
- Tenant scoping is enforced at the permission level.
- Scheme admins can access their company's data.
- Beneficiaries can only access their own records.
"""

from rest_framework import permissions

from apps.accounts.models import UserRole


class IsTenantMember(permissions.BasePermission):
    """
    Ensure user belongs to the same tenant as the requested resource.

    Views using this permission must implement `get_resource_company()` 
    or the resource must have a `company` attribute.
    """

    message = "You do not have access to this organization's data."

    def has_object_permission(self, request, view, obj):
        user = request.user

        # Must be authenticated
        if not user or not user.is_authenticated:
            return False

        # Get the resource's company
        resource_company = getattr(obj, "company", None)
        if resource_company is None:
            # If object has no company, allow (shouldn't happen for tenant-scoped models)
            return True

        return user.can_access_company(resource_company)

class IsOwnerOrSchemeAdmin(permissions.BasePermission):
    """
    Object-level permission: user owns the object or is a scheme admin.

    For Beneficiary objects, check if the requesting user is the beneficiary.
    For other objects, check if the user field matches.
    """

    message = "You can only access your own records."

    def has_object_permission(self, request, view, obj):
        user = request.user

        if not user or not user.is_authenticated:
            return False

        # Scheme admins can access all (within tenant)
        if user.role == UserRole.SCHEME_ADMIN:
            # Still check tenant access
            resource_company = getattr(obj, "company", None)
            if resource_company:
                return user.can_access_company(resource_company)
            return True

        # Beneficiaries can only access their own records
        if user.role == UserRole.BENEFICIARY:
            # Check if this is a Beneficiary object
            if hasattr(obj, "user"):
                return obj.user == user
            # Check if this is an allocation/event linked to a beneficiary
            if hasattr(obj, "beneficiary"):
                beneficiary = obj.beneficiary
                return beneficiary.user == user if beneficiary else False

        return False

"""
Custom exception handling for the REST API.

Provides consistent error response format and audit logging for errors.
"""

import logging

from django.core.exceptions import PermissionDenied, ValidationError as DjangoValidationError
from django.http import Http404
from rest_framework import status
from rest_framework.exceptions import APIException
from rest_framework.response import Response
from rest_framework.views import exception_handler

logger = logging.getLogger(__name__)


def custom_exception_handler(exc, context):
    """
    Custom exception handler that provides consistent error format.

    Response format:
    {
        "error": {
            "code": "error_code",
            "message": "Human-readable message",
            "details": {...}  # Optional additional details
        }
    }
    """
    # Call REST framework's default exception handler first
    response = exception_handler(exc, context)

    # Get request info for logging
    request = context.get("request")
    view = context.get("view")

    if response is not None:
        # Standardize the error format
        error_data = {
            "error": {
                "code": _get_error_code(exc),
                "message": _get_error_message(exc, response),
            }
        }

        # Add details if available
        if hasattr(exc, "detail") and isinstance(exc.detail, dict):
            error_data["error"]["details"] = exc.detail

        response.data = error_data

        # Log the error
        logger.warning(
            "API error: %s %s - %s (%s)",
            request.method if request else "?",
            request.path if request else "?",
            error_data["error"]["code"],
            error_data["error"]["message"],
        )
    else:
        # Handle Django exceptions not caught by DRF
        if isinstance(exc, Http404):
            error_data = {
                "error": {
                    "code": "not_found",
                    "message": "The requested resource was not found.",
                }
            }
            response = Response(error_data, status=status.HTTP_404_NOT_FOUND)
        elif isinstance(exc, PermissionDenied):
            error_data = {
                "error": {
                    "code": "permission_denied",
                    "message": "You do not have permission to perform this action.",
                }
            }
            response = Response(error_data, status=status.HTTP_403_FORBIDDEN)
        elif isinstance(exc, DjangoValidationError):
            error_data = {
                "error": {
                    "code": "validation_error",
                    "message": str(exc),
                }
            }
            response = Response(error_data, status=status.HTTP_400_BAD_REQUEST)
        else:
            # Log unexpected exceptions
            logger.exception(
                "Unhandled API exception: %s %s",
                request.method if request else "?",
                request.path if request else "?",
            )
            error_data = {
                "error": {
                    "code": "internal_error",
                    "message": "An unexpected error occurred.",
                }
            }
            response = Response(error_data, status=status.HTTP_500_INTERNAL_SERVER_ERROR)

    return response


def _get_error_code(exc):
    """Get a machine-readable error code from the exception."""
    if hasattr(exc, "default_code"):
        return exc.default_code

    # Map exception types to codes
    code_map = {
        "AuthenticationFailed": "authentication_failed",
        "NotAuthenticated": "not_authenticated",
        "PermissionDenied": "permission_denied",
        "NotFound": "not_found",
        "MethodNotAllowed": "method_not_allowed",
        "Throttled": "rate_limit_exceeded",
        "ValidationError": "validation_error",
    }

    exc_name = exc.__class__.__name__
    return code_map.get(exc_name, "error")


def _get_error_message(exc, response):
    """Get a human-readable error message."""
    if hasattr(exc, "detail"):
        if isinstance(exc.detail, str):
            return exc.detail
        elif isinstance(exc.detail, list):
            return exc.detail[0] if exc.detail else "An error occurred."
        elif isinstance(exc.detail, dict):
            # Get first error message from dict
            for key, value in exc.detail.items():
                if isinstance(value, list) and value:
                    return f"{key}: {value[0]}"
                elif isinstance(value, str):
                    return f"{key}: {value}"
            return "Validation error."

    return "An error occurred."
